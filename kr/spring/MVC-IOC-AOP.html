# MVC & IOC & AOP

# MVC

Spring MVC是一个基于MVC架构的用来简化web应用程序开发的应用开发框架，它是Spring的一个模块，都属于表现层的框架。在web模型中，MVC是一种很流行的框架，通过把Model，View，Controller分离，简化开发，减少出错，方便组内开发人员之间的配合

# Spring MVC 流程

  1. 一个 DispatcherServlet 前端控制器，是 SpringMVC 的核心，由它拦截并分发用户请求；
  2. 再通过处理器映射器根据 URL 定位对应的处理器执行链；处理器执行链包含拦截器和控制器，可在业务处理之前对请求进行拦截处理；
  3. DispatcherServlet 再通过处理器适配器处理具体的控制器业务方法；
  4. 控制器返回 ModelAndView 模型和视图至前端控制器，再通过视图解析器定位具体的视图资源并渲染数据返回前端。

# IOC

# IOC 原理/思想

  1. Spring IOC 即控制反转，是依赖倒置原则的一种代码设计思路，是由 Spring IOC 容器来负责对象的生命周期及对象之间的依赖关系。
  2. 一般情况下，通过 new 的方式创建对象并建立对象间的依赖关系，这种方式造成了对象间的强耦合，而 Spring IOC 通过依赖注入（DI）的方式实现了对象间的解耦。
  3. 依赖注入有三种实现方式，分别是构造方法注入、Setter 方法注入、接口注入

依赖倒置原则：上层模块不应该依赖底层模块，应该依赖于抽象；抽象不应该依赖于细节，细节应该依赖于抽象

# IOC 的好处

  1. 资源集中管理，实现资源的可配置和易管理。
  2. 降低了使用资源双方的依赖程度，即资源间的耦合度

# 依赖注入（DI）区别？

  1. 构造方法注入：在类的构造方法中声明所依赖的类，这种方式灵活性比较差，在对象初始化时，其所依赖的对象必须显示地注入。
  2. Setter 方法注入：通过 Setter 方法注入所依赖的对象，这种方式灵活性比较好，可动态改变依赖。
  3. 接口注入：一般情况下不使用接口注入，接口注入会造成被注入对象实现不必要的接口，带有侵入性。

# Spring IOC 初始化过程

  1. Spring IOC 容器是基于 BeanFactory 和 ApplicationContext 两个接口实现的。
  2. 资源定位：通过 XML 或注解定义好 Bean 配置信息后，Spring 通过 Resource 完成对Bean配置信息的定位
  3. 载入：将定位到的 Bean 配置信息解析成 BeanDefinition 对象
  4. 注册：将 BeanDefinition 对象统一注册到 IOC 容器的 Bean 定义注册表中（HashMap）

# IOC 容器初始化过程

  • IOC：控制反转是一种通过 XML 或注解来生成或获取 Bean 对象的技术。
  • IOC 容器：Spring 通过 IOC 容器管理 Bean 的生命周期及 Bean 间依赖关系。 IOC 容器实现一个顶级容器接口 BeanFactory

# IOC 容器初始化

在 SpringBoot 中，ApplicationContext 应用上下文（IOC 容器）的初始化过程：

protected ConfigurableApplicationContext createApplicationContext() {
    Class<?> contextClass = this.applicationContextClass;
    if (contextClass == null) {
        try {
            switch (this.webApplicationType) {  // 根据不同的应用类型创建不同的应用上下文
            case SERVLET:   // servlet 应用
                contextClass = Class.forName(DEFAULT_SERVLET_WEB_CONTEXT_CLASS);
                break;
            case REACTIVE:  // reactive 应用
                contextClass = Class.forName(DEFAULT_REACTIVE_WEB_CONTEXT_CLASS);
                break;
            default:
                contextClass = Class.forName(DEFAULT_CONTEXT_CLASS);
            }
        }
        catch (ClassNotFoundException ex) {
            throw new IllegalStateException(
                    "Unable create a default ApplicationContext, "
                            + "please specify an ApplicationContextClass",
                    ex);
        }
    }
    return (ConfigurableApplicationContext) BeanUtils.instantiateClass(contextClass);   // 反射调用构造方法实例化 IOC 容器
}

# Bean 对象装载

将配置的 Bean 对象初始化后装载进应用上下文（IOC 容器）中：

private void prepareContext(ConfigurableApplicationContext context,
        ConfigurableEnvironment environment, SpringApplicationRunListeners listeners,
        ApplicationArguments applicationArguments, Banner printedBanner) {
    context.setEnvironment(environment);
    postProcessApplicationContext(context);
    applyInitializers(context);
    listeners.contextPrepared(context);
    if (this.logStartupInfo) {
        logStartupInfo(context.getParent() == null);
        logStartupProfileInfo(context);
    }
    // 应用上下文中获取一个简单 IOC 容器
    ConfigurableListableBeanFactory beanFactory = context.getBeanFactory();
    beanFactory.registerSingleton("springApplicationArguments", applicationArguments);
    if (printedBanner != null) {
        beanFactory.registerSingleton("springBootBanner", printedBanner);
    }
    if (beanFactory instanceof DefaultListableBeanFactory) {
        ((DefaultListableBeanFactory) beanFactory)
                .setAllowBeanDefinitionOverriding(this.allowBeanDefinitionOverriding);
    }
    // 加载 @SpringBootApplication 声明的类
    Set<Object> sources = getAllSources();
    Assert.notEmpty(sources, "Sources must not be empty");
    load(context, sources.toArray(new Object[0]));
    listeners.contextLoaded(context);
}
protected void load(ApplicationContext context, Object[] sources) {
    if (logger.isDebugEnabled()) {
        logger.debug(
                "Loading source " + StringUtils.arrayToCommaDelimitedString(sources));
    }
    // BeanDefinition 加载器
    BeanDefinitionLoader loader = createBeanDefinitionLoader(
            getBeanDefinitionRegistry(context), sources);
    if (this.beanNameGenerator != null) {
        loader.setBeanNameGenerator(this.beanNameGenerator);
    }
    if (this.resourceLoader != null) {
        loader.setResourceLoader(this.resourceLoader);
    }
    if (this.environment != null) {
        loader.setEnvironment(this.environment);
    }
    loader.load();
}

# 循环依赖处理？

参考：Spring 循环依赖 (opens new window)

  1. 循环依赖其实就是循环引用，就是多个 bean 互相引用对方，最终形成一个引用闭环。
  2. 构造方法注入：调用构造方法初始化对象时，由于依赖对象的强制要求，当出现循环依赖时，由于依赖对象无法通过初始化拿到依赖对象的引用，Spring 只能抛出异常；
  3. Setter 方法注入：
    • 单例 Bean：Spring 在还没有完全初始化单例 Bean 时，会将 Bean 对象放入缓存中提前暴露该 Bean 对象，当其它对象需要依赖该 Bean 时，会从缓存中获取到该 Bean 的引用，通过 Setter 方法进行注入；
    • 原型 Bean：Spring容器不进行缓存，因此无法提前暴露一个创建中的原型 Bean，所以只能抛出异常。

# Bean 的作用域

  1. 单例：容器中仅存在一个 Bean 实例；
  2. 原型：每次从容器中获取 Bean 时，返回一个新的实例；
  3. Request：每次 HTTP 请求都创建一个新的 Bean 实例；
  4. Session：同一个 HTTP Session 共享一个 Bean 实例；
  5. Global Session：一般用于应用环境；
  6. request、session和global session三种作用域仅在基于web的应用中使用
  7. 有状态的bean都使用Prototype作用域，无状态的一般都使用singleton单例作用域

单例 Bean 是线程不安全的，解决方法：在类中定义一个 ThreadLocal 成员变量，将需要的可变成员变量保存在 ThreadLocal 中
