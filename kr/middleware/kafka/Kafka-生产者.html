# Kafka 生产者

# 消息对象 ProducerRecord

  • topic/partition 分别代表消息要发往的主题和分区号；headers 字段是消息的头部
  • key 是消息的键，它不仅是消息的附加信息，也用来在未指定分区时计算分区号进而可以让消息发往特定的分区
  • timestamp 消息时间戳，两种类型 CreateTime(消息创建时间)/LogAppendTime(消息追加到日志文件的时间)

# 消息发送模式

发送消息主要有三种模式：发后即忘（fire-and-forget）、同步（sync）及异步（async）

/**
 * 发后即忘：直接调用 send，不处理返回值，可靠性低
 * 同步：(未指定回调函数)，调用 send 返回值 Future 对象 get 方法，这时会阻塞等待消息发送成功/发送异常，可靠性高
 * 异步：通过指定回调函数来实现异步的消息发送确认：消息发送成功/发送异常
 * KafkaProducer 两种异常：可重试异常（消息发送异常时按规定次数进行重试）和不可重试异常
 */
public Future<RecordMetadata> send(ProducerRecord<K, V> record) {
    return this.send(record, (Callback)null);
}
public Future<RecordMetadata> send(ProducerRecord<K, V> record, Callback callback) {
    ProducerRecord<K, V> interceptedRecord = this.interceptors.onSend(record);
    return this.doSend(interceptedRecord, callback);
}

生产者发送消息至 broker 时，可能要经过 拦截器(Interceptor) > 序列化器(Serializer) > 分区器(Partitioner) 处理

// 多个拦截器组成的拦截链：按照 interceptor.classes 参数配置的拦截器顺序执行
public interface ProducerInterceptor<K, V> extends Configurable {
    // 将消息序列化和计算分区之前会调用生产者拦截器对消息进行拦截处理
    ProducerRecord<K, V> onSend(ProducerRecord<K, V> var1);
    // 在消息被应答之前或消息发送失败时调用生产者拦截器的 onAcknowledgement 方法，优先于用户设定的 Callback 之前执行
    void onAcknowledgement(RecordMetadata var1, Exception var2);
    void close();
}

/**
 * 生产者发送消息的默认分区分配策略（分区器的作用）：
 * 指定分区时，消息发送至指定的分区号
 * 未指定分区但有 key 时，采用 MurmurHash2 算法(具备高运算性能及低碰撞率)对 key 进行哈希，并与主题分区数取余得分区号（所有分区中任意一个）
 * 未指定分区且无 key 时，消息将会以轮询的方式发往主题内的各个可用分区（所有可用分区中任意一个）
 */
public int partition(String topic, Object key, byte[] keyBytes, Object value, byte[] valueBytes, Cluster cluster, int numPartitions) {
    return keyBytes == null ? this.stickyPartitionCache.partition(topic, cluster) : Utils.toPositive(Utils.murmur2(keyBytes)) % numPartitions;
}

# 生产者客户端架构

  • 主线程：消息经过拦截器/序列化器/分区器后缓存到消息累加器对应的 ProducerBatch（若无/不可写入则创建）
  • 发送线程：从消息累加器获取 ProducerBatch 后发送至对应的 broker，并负责 kafka 集群元数据更新
// 消息累加器，若消息生产速度超过发送速度，则消息累加器空间不足，KafkaProducer.send() 会阻塞/抛出异常
public final class RecordAccumulator {
    // 为主题的每个分区维护一个双端队列
    // 主线程写消息时，追加到双端队列的尾部；发送线程读取消息时，从双端队列的头部读取
    // ProducerBatch 为消息批次，包含一至多个 ProducerRecord，批量发送可减少网络请求的次数以提升整体的吞吐量
    private final ConcurrentMap<TopicPartition, Deque<ProducerBatch>> batches;
}

# ACK 应答机制

acks 参数配置：

  • acks=0：生产者不等待应答直接返回，当 broker 故障时可能丢失数据，但可达到最大吞吐量
  • acks=1：生产者等待 ack，leader 落盘成功后返回 ack，若 follower 同步成功前 leader 故障则会丢失数据，是消息可靠性和吞吐量之间的折中方案
  • acks=-1(all)：生产者等待 ack，ISR 中所有副本落盘成功后返回 ack，若 follower 同步完成后，broker 发送 ack 前，leader 故障则生产者可能重复投递造成数据重复

注意：acks=-1 时，可达到最强可靠性但不一定可靠，因 ISR 中可能只有 leader 副本，这就退化成了 acks=1 的情况

# Exactly Once 语义/幂等性

::: info 消息中间件的消息传输保障有 3 个层级：

  • At Least Once 语义：当 acks=-1 时，可保证生产者生产的消息投递成功，不会丢失，但无法保证数据不重复
  • At Most Once 语义：当 acks=0 时，可保证每条消息只会被发送一次，即保证数据不重复，但无法保证数据不丢失
  • Exactly Once 语义：数据不丢失也不重复 ::: 幂等性：指无论生产者发送多少重复数据，broker 端只会持久化一条数据；幂等性结合 At Least Once 语义构成 Exactly Once 语义，即
At Least Once + 幂等性 = Exactly Once

将生产者参数 enable.idompotence=true 即可开启幂等性；

Kafka 幂等性，在底层设计架构中引入了两个概念：

  • ProducerID：在每个新 Producer 初始化时，分配一个唯一的 ProducerID，标识本次会话，这个 ProducerID 对客户端使用者是不可见的
  • SequenceNumber：对于每个 ProducerID，Producer 发送数据的每个 Topic/Partition 都对应一个从 0 开始单调递增的 SequenceNumber 值

每条消息中附带了 PID/SequenceNumber，相同 PID/SequenceNumber 消息只会持久化一次，确保数据不重复

注意：由于 PID 重启会发生变化，且 SequenceNumber 是针对分区而言，故幂等性无法保证跨分区跨会话的幂等
